## Day 35: Advanced Tomcat Configurations & End-to-End Java CI/CD Pipeline (Apr 28, 2026)

### 🐱 1. Advanced Tomcat Administration
Before automating, we mastered managing multiple Tomcat instances on a single VM.

**A. Running Multiple Instances of Tomcat:**
If you run two Tomcats, their ports will collide and crash.
- Extracted a fresh Tomcat tarball into `/opt/tomcat2`.
- Edited `/opt/tomcat2/conf/server.xml` to change the default ports to avoid conflicts:
  - **Connector Port:** Changed from `8080` to `8081`.
  - **Shutdown Port:** Changed from `8005` to `8006`.
- Edited `conf/tomcat-users.xml` to add `manager-script` and `manager-gui` user roles for remote deployment access.

**B. The `ROOT.war` Magic:**
By default, placing `my-app.war` into `webapps/` means you must access it via `http://IP:8080/my-app`. 
- **The Fix:** We renamed our application to `ROOT.war` and placed it in the `webapps/` directory.
- Tomcat automatically replaces its default homepage, making our Java application accessible directly at the root URL: `http://IP:8080/`.

### 🔄 2. Continuous Integration (Job 1: Build & Package)
Created a Jenkins Freestyle Job named `Java-app` to handle the CI process.
- **Source Code Management:** Linked the GitHub repository containing the `src/` folder and `pom.xml`.
- **Build Steps:** Configured the Maven integration.
  - Specified the exact path to the `pom.xml`.
  - Executed Maven Goals: `clean package`.
- **Post-Build Actions (Archiving Artifacts):**
  - Instead of leaving the compiled `.war` file hidden inside the Jenkins workspace, we used the **Archive the Artifacts** action. This makes the `.war` file visible and downloadable directly on the Jenkins Job Dashboard.

### 🚀 3. Continuous Deployment (Job 2: Deploy to Tomcat)
Created a second Freestyle Job specifically for deployment (CD).

**A. Required Plugins Installed:**
1. **Copy Artifact Plugin:** Allows Job 2 to grab the `.war` file generated by Job 1.
2. **Deploy to container Plugin:** Allows Jenkins to push the `.war` file over the network to the Tomcat server.

**B. Job Configuration:**
- **Build Step:** Used *Copy artifacts from another project* to pull the `.war` file from the `Java-app` job.
- **Post-Build Action:** Used *Deploy war/ear to a container*.
  - Configured the Tomcat 11 Manager credentials (from `tomcat-users.xml`).
  - Set the Tomcat URL (`http://<VM_IP>:8080`).

### 🔗 4. Pipeline Chaining (The Automation Magic)
To achieve true CI/CD, these two jobs must run in sequence automatically.
- Went back to Job 1 (`Java-app`).
- Added a Post-Build Action: **Build other projects**.
- Selected Job 2 and set the condition: **Trigger only if build is stable**.
- *Result:* When new code is pushed to GitHub, Job 1 builds the `.war` file. If successful, it triggers Job 2, which copies the artifact and deploys it live to Tomcat without a single human click!
